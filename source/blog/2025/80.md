# 使用 Google Antigravity 在大地上码代码码...

使用 Google Antigravity 在大地上码代码码文字。

我刚从 Cursor 切换到 Google Antigravity，编程体验直接上了一个台阶，如果没有其他变化，我想我不会再切回去使用 Cursor 了。下面我介绍一下我如何使用这个编码怪兽。

首先，Anti（Antigravity 简写，以下同）的工作方式是分空间的，这也很好理解，空间相当于它的活动范围，在这个范围之内的文件，我只需要用艾特符号提及，它就知道我说的是哪一个。在右上角有一个 Agent Manager，这里面有许多空间，也可以自己创建，一个 Workspace 可以容纳多个项目，这些项目相互之间也可以像一个项目一样亲密无间，在 Anti 的眼里，它们没有差别。在～/.gemini 目录下，有以项目名称 + profile 分隔的工作记录，稍后我谈到的三大工件，它们也在这里。在这里的目录划分可以看出，在一个 Workspace 内部，Anti 对项目也是有划分的，它们既是一个整体，也是各自为政的。

其次，说一下最重要的两个配置。在左下角有一个 Antigravity Settings，点击它，里面有两个选项，是新手特别关心的：1，Auto Execution，代表脚本自己执行，我选择 Always Proceed；2，Review Policy，代表代码改动的审核策略，我选择的仍然是 Always Proceed。为什么对终端指令执行和代码改动，我都选择放任不管？因为大多数时候 AI 比我们聪明，更重要的是比我快，我还没有把审核内容看完，它可能就把活干完了。我在使用 Cursor 的时候基本也是放任不管，使用 Anti 更不可能管制了。有人可能担心会出 Bug，不用担心，有 Bug 或不足之处，再让 Anti 修改就是了。

最后，我说一下 Anti 的三大工件。Artifacts，在 Anti 内部叫工件，是 Google 工程师的发明，指的是在 AI 动作过程中不同部件之间相互协作所用的介质。最重要的三大件指的是： Implementation Plan，执行计划；Task、任务列表；Workthrough，执行总结。三者之中，Task 最简洁，它就是一个无序列表，干完一个前面打一个勾。Implementation Plan 最复杂，它列举了详细的执行计划。最后，Anti 干完活了，还会给一个执行总结（Workthrough），AI 在认知上一般很容易产生幻觉，在行动上也容易像醉汉一样脚步凌乱，Anti 为了显示自己清醒，一定要给一个执行总结出来：看，我干得不错吧，知行合一了！

现在 AI 已经发展到了目录化阶段，各大厂商 IDE 都会在目录中存储与工作空间、项目有关的文件。在此之前，由 Claude 带头执行的，是提示语的结构化。还有更早的，由 GPT 带头执行的身份标识化，那个时候写提示语，先要给 AI 一个身份：你是一位什么什么的厉害人物。AI IDE 发展到现在，身份化已经不重要了，甚至结构化都不重要了（你在描述你的需求时，用不用繁琐的结构结果都是一样的），现在最重要的，也是一直都重要的，就是逻辑清晰、无漏洞。我们的逻辑清晰严谨，AI 的执行结果就漂亮、无懈可击；我们的逻辑一塌糊涂，漏洞百出，甚至自相矛盾，AI 的执行结果就惨不忍睹。AI 的表现，完全掌握在我们手里。

bye，我暂时对 Anti 体验只有这么多，以后有机会再分享。
